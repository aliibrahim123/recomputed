Template Statements, Above Revolutionary 27-9-2025

Template Evolution in the Web

Template Files

from the beginning of the web, and of the UI Design in general, there exist the need to link the UI markup with its logic and states, making them coexist in the same realm.

having the dream of declaring ui in a high level way, the original design paradigm were existing in the other side, the options were writing in verbose OOP style, or in low level draw call style, with manually retriving and updating ui on changes, or having a bare primitives of basic binding.

as the time went on and the ambitions grow more, the need for true solution for UI logic linking grow more aperant, so template files and there respective framewarks begined rising.

template files are a big blobs of reusable UI chunks, written in their native HTML form, with specific attributes and content sections coming from embeded expressions that reevaluate when updates occurs implicitly.

while template file was revolutionary in every mean, they had alot of problems, mainly the remaining separation of markup from its logic, and having inputs comming from the global component scope, which created countless hip hoping between files.

Template Expressions

the next steps in evolution the web took were to bring togather markup with its logic, and this started primary with react and its JSX expressions.

template expressions, despite their countless variations, had one existential goal, to be inlinable inside the logic, both locational and relational.

template expression are from the name, expressions or try to be like that, they are like template files chunks of UI written in its natural form, having embedded expressions that reevaluate on updates, however, template expressions can access the scope of its host logic, which eliminate the need of a middle layer and provide better encapsulation and type safety.

there is a reason for why the new frameworks you find in the wild is using template expressions, from Angular to React to Solid to Lit, while the syntax and its form change from framework to another, the core idea remains the same, if you used tagged templates or you used custom syntax.

template expressions formed the bases of a new era, the era of the modern reactive web we know and live in, an era feauring limitless reactivity and unmatched declarativity. yet from nowhere, someone had come with an idea that made the peak of declarativity look verbose.

Template Statements

lately, Dominic Gannaway (auther of Inferno), had come up with a new framework that will make Ripples in the web world.

the revolutionary feature the ripple framework invented is template statements, apart from the unusual syntax you can notice in the example, you can see that html structure is the main structure in the component, and the logic in defined normally inside the html structure, statements not expressions, all of this inside a javascript file, this is the template statements.

template expressions are html structure inside the logic that acts as an expression, they returns ui structure. they can have as much embeded expression as they want, but at the end of the day, they are limited only to a subset of expressions, not decleration and statements.

template statments on the other hand act like normal statements, they dont return structure, each one define a section of the structure and togather they form the entire structure, like template expression, they are embeded inside the logic and can access their scrope, however, they can host there own localized logic and other template statements, they can have all the javascript features (variables, statements, functions, classes).

how much we can go, it depends on the framework, but theorically, we are using javascript, so this is possible.

The Power of Template Statements

this paradigm in ui definition has outstanding abilities, primary the ability to directly inline markup with its logic, with encapsulation, at any level, as template statements are statements, they can host their own logic in a separate lexical scope.

in the past approuches, you only have one scope per component for logic markup state sharing, while in template statements, you can split your component into smaller sections, each with its own logic and scope, and repeat this step at each level.

comparison between template expressions and template statements

In addition, being multiple chunks not singleton blobs, template statements can use regular control flows (if, for, while, switch, etc.), and dont reinvent the wheel in every frameworks, removing the need for directives, utility components, mixins, or whatever feature used to hack conditional and list rendering.

NeoComp implementation

neocomp adopted the template statements paradigm, but with phelosophy oposite to Ripple, ripple adds new syntax to javascript, and relies on complex compilation step, things that violates neocomp core phelosophy in simple vanilla javascript enhanced with optional simple transformations.

templating in neocomp is now composed of multiple fragments of ui called chunks, each chunk is defined around its logic in regular html syntax.

each chunk is formed of multiple part splitable at any point, defined around its own logic through $temp tagged template literals, state can be set / binded to attributes and content sections through ${exp} embeded expressions.

chunks in their own doesnt bring noticable new features to the table, however, when combined with javascript features, the powers of template statments starts to shine, as chunks are defined with vanilla javascript.

effects of template statements on neocomp

other than the previously discussed features of template statements, from localized state markup coexisting to native control flow usage, template statements just made the biggest refactor to neocomp ever.

primary, it drastically simplified the templating internals, what was before a series of reasonable size systems for generation, walking, actions, and construction had truned into just a single function with some helper functions found in a single file.

this reduces the file size 25% (from 10kb to 7.5kb ish gzip), while chunks added many features, it also simplified the logic through deduplications and removing the need of a middle layer.

in addition, since of the localized logic the chunks bringed, the need for global state with naming support was gone, which enhanced the state management with localized state and deep state reactivity.

template statements single handedly transformed the templating experince in neocomp, from old days template files with modern features, to multilevel chunked localized coexisting template statements that rivals the modern frameworks with super flixibility and efficiency.

template statments were the needed forth pillar of neocomp, its flixibility complements the effeciency of fine grained reactivity, the organization of object oriented components, and the added performance of native transformations, further supporting neocomp core paradigm, native from ui design with direct implicit bindings drived by imperative construction, the paradigm for static structure dynamic content, the structure of the web.

and dont forget, this is at the end of the day just a concept framework, a case study for an opinionated view on simplicity in ui design, simplicity in directness outperforms simplicity in declarativeness, we dont need to eliminate the last character in the source, we only need implicit direct bindings in our inlined html templates.
