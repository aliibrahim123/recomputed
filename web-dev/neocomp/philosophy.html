NeoComp Philosophy 1-12-2025

NeoComp, depsite sharing a ton of common points between the other reactive web framework, it has a different and unique philosophy that sets it in an opposite corner of the spectrum, and it is derived from a simple idea.

Conservative Simplicity

conservative simplicity is a direct and lightweight simplicity in api surface and design, simplicity that is conservative in its implementation complexity.

it is simplicity that does not seek to be the state of the art in deleting the last character in the source, at the cost of the implementation depth, instead, it is simplicity summerized in a collection of utility functions, having a transformative impact.

it is not about barebone imperative over rich declarative, in essence, it is lightweight declarative definitions / actions inlined within imperative flow, simplifing the tedious work while being direct and effecient.

conservative simplicity as an ideology in web frameworks flips the table upside down, it creates a unique framework that doesnt obey the modern mentality of reactive declarativity ui design.

all the muscle memory you aquired and the design patterns you used will need to drop while using neocomp, however, new and wonderful patterns will replace in the longrun.

and now after establishing the core ideology, lets dive into the indevidual rules.

Fixed Reactive Struture

neocomp, like other fine grained reactivity frameworks, create the full structure of ui at initialization, with the bindings infered from the templates, then the updates flow directly between states and effects to the DOM.

this is the reverse of what other frameworks do, notobly the VDOM powered ones, where on each update, the relivent render functions rerun creating fresh structure, then the reconciling mechanisms continue the work.

this approuch enable neglishable cost updates at the cost of fixed struture, while seeming like restriction, in reality it is nothing like that, as all the HTML powered pages are fixed structure with dynamic content and navigations, structures that doent change from construction, can be hidden and if they changes their identity is what changed.

these strucutres doesnt need to be mathimatical formula, Treating them as such created problems that didn't need to exist, for benefits that were ultimately not important.

chunked construction

as contrary to the templating approuch taken by the majority, defining the entire structure of the component as a big blob at the end, neocomp has a unique way in construction.

in neocomp, templates are created as multiple chunks in an imperative flow, each section is defined around its logic, and construction is not restricted to single location.

this feature singly enables a lot of capabilities, it remove most restrictions on templates, localize logic to the heart, and let many outstanding pattern to emerge.

you can use regular control flow, extract and localize major and minor sections into their own place and scope, create reusable nano fragments, and countless more, you are free in structuring your own templates.

Component Own Scope

functional components while having less verbosity, they have a limited scope, there logic, structure and local utilities share the same place, which make things crowd quickly

neocomp components on the other hand are defined in classes, having there own scope, which makes things much more manageable, as you can freely grow vertically without the fear of 60 lines functions.

when the component grow, you can migrate major sections into their own functions, with shared states in fields, independent logic inside methods, and local utilities hosted within the class, all under the same curly braces.

you should not afraid large components as classes can host them easily, moreover, splitting them into smaller component are like extracting major sections into their own methods plus acward state sharing.

Components as Objects

components in neocomp are not abstract concepts, they are real objects that can interact and be interacted with, like regular dom elements.

components are not just ui factories, they are ui entities that can be interacted with, they can expose a public api inside public methods, fields and events.

the comunications between components are not limited to static parent to child data flow or hacked grandfather grandchildren state elevators, any 2 components can interact togather at any time through the standared mechanism of events, methods and signals.

no mini components

mini components, components of few lines, are bad in neocomp, other than writing relatively large bolerplate with respect to the code size, components has there overhead that is larger than a logic of a component having small structure.

instead, you should create independent sections not entire components, functions that create a chunk using the component resources, having significally lower bolerplate and coxisting like a part of a component.

render and hide

in render function frameworks, conditional rendering is done by conditionally recreating and destroying the structure and components, this approuch has a flaw of rerendering the whole structure every time, which is expensive.

instead, and the approuch used by neocomp and similar frameworks, you can create the structure once on creation, and conditionally showing it on demand, a small word change fixing the performance of many pages.

DOM Ownership

neocomp, opposite to the majority of the frameworks, give you full owernership of the DOM, its actions are localized to specific elements and attributes defined in the template, you can laterialy move the elements around and the bindings remains working, even cross documents boundaries.

while manual dom manipulation is the target frameworks from their preciption was trying to remove, having a reliable control on the DOM is important for the existence of some elements like dropdown and models.

another feature neocomp has is wrapping existing elements and owning them, components can be passed an element so that they add functionality to their contents, a feature nice to have.

vanilla transformations

neocomp is a vanilla javascript framework, it doesnt add new syntax to the language opposite to what all the frameworks do, it uses old school string templates literials like lit.js but with a twist.

the template strings can be optimized at build time into effecient form, removing the need of parsing the templates at runtime, this step is optional as the source code can be run normally without a build step, it is just a small touch of optimization.

neocomp is a unique framework with a unique simplicity, simplicity that merges between declarativity and imperativity in an effecient and lightweight way.

it is a vanilla javascript framework with a new paradigm: declarative html templates with implicit direct bindings, derived by imperative chunked construction, optionally enhanced with vanilla transformations.

taking the best of the past frameworks, neocomp present to you a case study how bringing back old concepts in a modern look, mixing them with the best of the new, while being conservative in the amount of used magic, can lead to a truely powerful yet simple framework.
